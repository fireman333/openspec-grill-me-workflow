① OpenSpec 原生開發循環

OpenSpec 把每一個 feature / 重構 / 變更包成一個 change proposal，走 propose → tasks → implement → archive 四段。底層由 openspec CLI + /opsx:* plugin 命令提供。

痛點：openspec 預設假設你已經知道「要 propose 什麼」。在 vibe-coding / 自由開發場景，最常卡住的是 propose 之前的概念釐清 跟 跨 session 的 context 丟失。

② /spec 5 個 gate 套在外面

/spec skill 是 cross-session wrapper，補的是 lifecycle 邊界（init / resume / handoff）跟 歷史回填（retro）；不重複實作 propose / apply / archive。

③ grill-me 兩個切入時機

切入點 A：/spec init 之前 — 用 /spec clarify

/spec clarify 就是 grill-me 的 thin wrapper：強制 Deep 模式 + software-project facet pack，輸出寫到 openspec/project-clarifications.md，/spec init 啟動時會讀進來當 project context。

15+ 題 recursive，~15–30 min。覆蓋 problem statement、users、scope boundary、non-goals、success criteria、constraints、prior art (no-wheels)、risks。

切入點 B：opsx:propose 之前 — 用 /grill quick

每要開一個 change proposal 之前先 grill 一輪，把模糊概念逼出可寫 spec 的形狀。

5–7 題 ~5 min。聚焦在這個 change 的 scope / acceptance criteria / 跟既有 spec 的衝突。

Hard rule（CLAUDE.md plan-phase）：偵測到「我想做一個 X」「規劃／設計／重構 Y」這類 planning signal，先 propose /grill quick 並等回應，不要直接進 plan mode、不要直接開寫 proposal、不要 spawn agent。使用者說「不用 grill、直接做」才略過。

⑥ 實際命令 sequence（複製即用）

場景 A：從零開新專案

cd ~/projects/my-new-app/
/spec assess              # 確認適合 openspec
/spec clarify             # ← Deep grill，~20 min，產出 clarifications.md
/spec init                # 讀 clarifications，建 openspec/ 結構

# 開第一個 change
/grill quick              # ← 5 min 釐清這次要做什麼
/opsx:propose user-auth   # 寫 proposal
/opsx:tasks               # 拆 tasks.md
# ... 實作 ...
/opsx:apply               # 更新 specs/
/opsx:archive             # 歸檔

場景 B：legacy repo 接手

cd ~/projects/legacy-app/
/spec assess
/spec init
/spec retro               # 從 git history 回填現有 specs/
# 之後同場景 A 的「開新 change」段

場景 C：新 session 接續

cd ~/projects/my-new-app/
/spec resume              # 自動讀 clarifications + active changes + last handoff
# 直接進「開新 change」段

⑦ 為什麼要這樣切

  • grill-me Deep 跟 OpenSpec project context 是同一件事：兩邊都在問「這專案要幹嘛、給誰用、邊界在哪、success 怎麼定義」。直接讓 grill 輸出餵給 init，不重複問。
  • grill-me Quick 跟 OpenSpec change proposal 是同一個顆粒度：一個 change = 一個 PR-sized 的範圍變動，5–7 題剛好。寫 proposal 之前 grill，比寫到一半發現 scope 不對再砍掉便宜。
  • plan-phase hard rule 自然觸發：「想做 X」「規劃 Y」這種訊號出現 → propose /grill quick 等回應 → grill 完寫 propose，整條鏈就是 spec-driven 的正規流程。
  • 不要在 resume / handoff 加 grill：那兩個 gate 是 lifecycle 機制不是釐清節點，硬加會讓接手變慢。
